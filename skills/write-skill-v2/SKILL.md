# Write-Skill V2.1 写作方法论

## 版本演进

**V1.0** → 高启示率、情感钩子、口语化表达  
**V2.0** → 增加信息密度控制、结构化拆解、记忆点设计  
**V2.1** → 基于与顶尖技术写手对比，优化：首句法则、数据精确性、具体化表达

---

## V2.1 核心升级（基于对比分析）

### 1. 首句法则（3秒入题）

**❌ V2.0 问题**: 开篇铺垫，3段后才给核心信息
```
"昨晚，DeepSeek 悄悄放出了 V4 预览版。
没有发布会，没有预热..."
```

**✅ V2.1 改进**: 首句必须包含核心事实
```
"今天上午，「DeepSeek-V4」 发布并开源。
DeepSeek V4 分两档：Pro 1.6T/49B、Flash 284B/13B，都支持 1M 上下文。"
```

**公式**: 时间 + 核心事实 + 关键数据（1句话）

---

### 2. 数据精确性（不用口语化折扣）

**❌ V2.0 问题**: 口语化表达，缺乏精确性
```
"缓存读取：0.1x（打9折）"
"便宜了一个数量级"
```

**✅ V2.1 改进**: 精确数字 + 单位 + 对比
```
"V4-Pro 输入缓存命中 1 元／百万 token，未命中 12 元／百万 token，输出 24 元／百万 token
对比 Claude 3 Opus 的 200K 上下文（约 15 元/百万 token）"
```

**原则**: 技术读者信任精确数字，不信任"大概"

---

### 3. 段落极简（1-2句话原则）

**❌ V2.0 问题**: 段落较长，信息堆叠
```
"缓存读取成本是基础输入价格的 0.1 倍——每个缓存 token 打 9 折。 
缓存写入成本 1.25 倍，存储 KV 张量有 25% 溢价。 
延长一小时缓存成本 2.0 倍。"
```

**✅ V2.1 改进**: 每段1-2句话，视觉分隔
```
"V4-Pro 输入缓存命中 1 元／百万 token，未命中 12 元／百万 token，输出 24 元／百万 token。

V4-Flash 对应 0.2 元／1 元／2 元。

定价表附了一行小字：受限于高端算力，目前 Pro 的服务吞吐十分有限，预计下半年昇腾 950 超节点批量上市后，Pro 的价格会大幅下调。"
```

**原则**: 短段落 = 易读 = 愿意读完

---

### 4. 信息性小标题（不用序号）

**❌ V2.0 问题**: 用"01 02 03"，信息量少
```
01 现象/问题
02 核心数据
03 技术原理
```

**✅ V2.1 改进**: 小标题本身就是信息
```
能干什么
1M 成了标配
架构三件套
训练账
后训练
成绩单：V4-Pro-Max 站到哪一档
真实任务上的表现
API 怎么用
能现在用
```

**原则**: 读者可以"扫标题"获取全文框架

---

### 5. 具体化金句（不用抽象概念）

**❌ V2.0 问题**: 金句偏抽象
```
"长文本不是目的，是基础设施"
```

**✅ V2.1 改进**: 金句用具体例子/类比
```
"对于一百万 token 来说，一次性塞入《三体》三部曲，还绰绰有余"
"这种分档思路，类似 Claude 的 Sonnet/Opus、GPT 的 Mini/Pro"
```

**原则**: 具体例子 > 抽象概念，读者记得住

---

### 6. 列表优先（不用长段落）

**❌ V2.0 问题**: 用段落列举
```
"缓存读取成本是基础输入价格的 0.1 倍...缓存写入成本 1.25 倍...延长一小时缓存成本 2.0 倍..."
```

**✅ V2.1 改进**: 用列表呈现
```
- Pro 1.6T 总参、49B 激活
- Flash 284B 总参、13B 激活
- 两档都支持 1M token 上下文
- 都开源，并给出了技术报告
```

**原则**: 列表 = 易扫读 = 信息密度高

## V2.1 写作流程

### Step 1: 首句设计（5分钟）
- 时间（具体：今天上午/昨天/2026年4月）
- 核心事实（产品发布/数据公布/政策出台）
- 关键数据（1-2个最重要的数字）

### Step 2: 框架搭建（10分钟）
- 列出 8-12 个小标题
- 每个小标题本身就是信息点
- 按"发布-能力-技术-数据-使用"逻辑排列

### Step 3: 内容填充（40分钟）
- 每段 1-2 句话
- 多用列表（- 或 1. 2. 3.）
- 数据精确到单位
- 每 300 字插入一个金句/引用块

### Step 4: 金句植入（10分钟）
- 检查是否有抽象概念，换成具体例子
- 用《三体》、Claude、GPT 等读者熟悉的参照物
- 确保每 500 字有一个"可传播句子"

### Step 5: 视觉优化（5分钟）
- 检查段落长度（>3句的拆分）
- 检查列表使用（能用列表不用段落）
- 检查小标题信息密度

---

## V2.1 检查清单

- [ ] 首句包含：时间 + 核心事实 + 关键数据
- [ ] 所有数据精确到单位（元/百万token，不是"打9折"）
- [ ] 每段不超过2句话
- [ ] 小标题不用"01 02"，用信息性标题
- [ ] 每300字一个金句（具体例子，不是抽象概念）
- [ ] 能用列表的地方不用段落
- [ ] 包含至少1个对比表格
- [ ] 包含至少1个步骤清单
- [ ] 结尾有明确的"怎么用"指南

## V2.1 vs V2.0 vs V1.0 对比

### 标题
```
V1.0: "22岁开出5万美元支票：没钱也能做天使投资？"
V2.0: "22岁天使投资人揭秘：5万美元起投，0资金门槛的3条路径"
V2.1: "DeepSeek V4 发布，全网最细解读 & 技术报告拆解"
       （明确承诺：最细解读、技术报告拆解）
```

### 开篇
```
V1.0: 故事铺垫，3段入题
V2.0: 场景切入，2段入题  
V2.1: 首句给事实，"今天上午，DeepSeek-V4 发布并开源"
```

### 数据
```
V1.0: "打9折、便宜一个数量级"（口语化）
V2.0: "0.1x基础价格"（半精确）
V2.1: "1元/百万token，对比Claude 15元/百万token"（精确+对比）
```

### 段落
```
V1.0: 长段落，5-8句话
V2.0: 中等段落，3-5句话
V2.1: 短段落，1-2句话，多用列表
```

### 金句
```
V1.0: "关键不是你有没有钱，是你有没有价值"（抽象）
V2.0: "长文本不是目的，是基础设施"（抽象）
V2.1: "一次性塞入《三体》三部曲，还绰绰有余"（具体）
```

## 适用场景

**最适合**:
- 技术产品发布解读
- 技术报告拆解
- 工具/框架教程
- 行业数据分析

**不适合**:
- 个人成长故事（用卡兹克风格）
- 情感类内容（用卡兹克风格）
- 快讯短评（用V1.0）
